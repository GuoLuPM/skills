# CodeCheck 作战手册

## 目标

这份参考手册提供一套通用的 CodeCheck 处理方法，重点不是解释某个具体平台，而是指导代理：

- 如何理解导出单；
- 如何区分“机械整改”和“语义风险整改”；
- 如何在不大改业务逻辑的前提下，提高过门禁效率。

## 一条核心原则

导出单是快照，不是事实本身。

先看当前代码，再看导出单。

不要把 Excel 行号当成真相，也不要把旧问题单当成当前状态。

如果是华为 `CodeArts Check` 语境，还要再加一条：

- 先分清产品事实、样例做法和当前代码事实。
- 华为特定规则名和产品配置问题，优先参考 `huawei-codearts-check.md`。

## 历史经验的稳定模式

### 1. 第一轮最应该先吃低风险机械规则

通常收益最高的是：

- 文件头
- docstring
- 长行
- 未使用导入
- 注释废代码
- `staticmethod`
- 纯方法顺序重排

原因：

- 数量大；
- 风险低；
- 可并行；
- 清完后，剩余问题的真实结构更清晰。

### 2. 后期剩余问题通常不是“更少”，而是“更危险”

多轮整改后，最常残留的是：

- `G.CLS.06`
- `G.CLS.11`
- `R0914`
- 返回值一致性
- 异常处理和日志规则
- 复杂度规则

这意味着：

- 不要因为剩余问题少，就误判为“最后一轮很轻松”。
- 后期要更保守，而不是更激进。

### 3. `G.CLS.06` 往往会成为后期主战场

这类规则经常还能通过“只搬方法块”解决。

处理时要坚持：

- 不改函数体；
- 不改签名；
- 不改装饰器；
- 不改返回值；
- 不借机做逻辑整理。

### 4. 有些规则看起来像风格问题，实际上是语义问题

尤其是：

- `protected-access`
- 返回值统一
- `try/except`
- `print -> logger`
- `SystemExit`
- `shell=True / shell=False`
- fallback / cache / history / runtime state

这些默认应该进“风险桶”，而不是“直接改”。

### 5. 可以为了过规范抽局部 helper，但不要借机重构

当规则抱怨：

- lambda 太长
- 局部变量太多
- 条件太多

优先考虑：

- 局部 helper
- 局部嵌套函数
- 中间变量归类

但仍然要保持：

- 调用链不变
- 返回值不变
- 异常边界不变

## 通用分桶法

### 低风险机械桶

- 文件头
- docstring
- 长行
- 未使用导入
- 注释废代码
- `staticmethod / classmethod`
- `G.CLS.06`
- 基础格式规则

### 中风险结构桶

- 局部 helper
- 局部嵌套函数
- 轻量命名收口
- 轻量返回整理

### 高风险语义桶

- `G.CLS.11`
- `G.LOG.02`
- `G.ERR.07`
- `G.ERR.11`
- fallback / cache bypass
- session / history / runtime scope
- 任何会改返回契约的规则

## 推荐执行顺序

1. 解析导出单，统计规则和文件分布。
2. 先按风险分桶。
3. 只把低风险桶并行派发。
4. 主代理统一复核。
5. 第一轮结束后重新统计剩余问题。
6. 若后期主要剩 `G.CLS.06`，单开“纯方法重排轮”。
7. 若后期主要剩高风险规则，停止自动推进，改成风险确认。

## 推荐验证闭环

每批次都固定做：

- `python -m py_compile <files>`
- `git diff --check -- <files>`
- 最小 import smoke
- 长行扫描

不要因为“只是规范”就跳过验证。

## subagent 协作建议

### 研究型

只回答：

- 哪些规则是纯表达规范；
- 哪些规则已经触碰语义边界；
- 哪些文件可以互不干扰地并行改。

### 实现型

只改指定白名单文件，只处理指定规则桶。

### 主代理

负责：

- 维护总清单；
- 复核 patch；
- 统一验证；
- 判断是否停手。

## 常见误区

### 误区 1

为了规范去顺手重构。

### 误区 2

看到 `R0914` 就直接拆大函数。

### 误区 3

旧 Excel 里有问题，就默认当前还存在。

### 误区 4

剩余问题少，就默认最后一轮更容易。

## 最后记住的 6 条

1. 当前代码永远优先于导出单。
2. 第一轮先打低风险机械规则。
3. 后期剩余问题更接近语义边界。
4. `G.CLS.06` 可以单独开轮处理。
5. 高风险规则默认只记风险，不自动整改。
6. 这份技能的价值是给代理稳定流程，而不是堆产品背景介绍。
