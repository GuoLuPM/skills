# CodeCheck 作战手册

## 目标

这份参考手册提供一套通用的 CodeCheck 处理方法，重点不是解释某个具体平台，而是指导代理：

- 如何理解导出单；
- 如何区分“代码表达形式整改”和“超出表达形式边界的问题”；
- 如何在不改业务逻辑、不做代码重构的前提下，提高过门禁效率。

## 一条铁律

CodeCheck 只处理代码表达形式。

不要用 CodeCheck 的名义去做：

- 业务逻辑修改
- 代码重构
- 架构重构

## 一条核心原则

导出单是快照，不是事实本身。

先看当前代码，再看导出单。

不要把 Excel 行号当成真相，也不要把旧问题单当成当前状态。

多轮导出单要按“当前轮优先”处理。旧导出单中的 `G.CLS.06` 方法顺序关系可能和新导出单形成环，不能要求所有历史表同时为 0。

如果是华为 `CodeArts Check` 语境，还要再加一条：

- 先分清产品事实、样例做法和当前代码事实。
- 华为特定规则名和产品配置问题，优先参考 `huawei-codearts-check.md`。

## 历史经验的稳定模式

### 1. 第一轮最应该先吃低风险机械规则

通常收益最高的是：

- 文件头
- docstring
- 长行
- 未使用导入
- 注释废代码
- `staticmethod`
- 纯方法顺序重排

原因：

- 数量大；
- 风险低；
- 可并行；
- 清完后，剩余问题的真实结构更清晰。

### 2. 后期剩余问题通常不是“更少”，而是“更危险”

多轮整改后，最常残留的是：

- `G.CLS.06`
- `G.CLS.11`
- `R0914`
- 返回值一致性
- 异常处理和日志规则
- 复杂度规则

这意味着：

- 不要因为剩余问题少，就误判为“最后一轮很轻松”。
- 后期要更保守，而不是更激进。

### 3. `G.CLS.06` 往往会成为后期主战场

这类规则经常还能通过“只搬方法块”解决。

处理时要坚持：

- 不改函数体；
- 不改签名；
- 不改装饰器；
- 不改返回值；
- 不借机做逻辑整理。

不要猜全局排序规则，例如“公有在前”“私有在前”“字母序”。优先从导出单问题描述解析精确关系：

- `A should be after B` 表示边 `B -> A`。
- 按文件和类分组建 DAG。
- 对类内方法做拓扑排序。
- 只替换 `FunctionDef/AsyncFunctionDef` 在类体中的槽位，其他语句、常量、内部类保持原位。
- 若出现缺失方法、重复方法名或环，停止自动改，先报告。

重排前后必须做 AST 级保护：

- 记录每个方法的 AST hash。
- 记录每个方法的装饰器列表。
- 重排后逐个对比。
- hash 或装饰器变化不为 0，就不能声称“只改顺序”。

处理完成后再按当前导出单逐条验证 `A should be after B`，失败数必须为 0。

### 4. JS/ECMAScript 大批量问题要用 AST，不要用正则硬改

常见低风险表达规则：

- `G.EXP.03`：嵌套三元表达式。
- `G.DCL.01`：禁止 `var`。
- `G.DCL.03`：每个语句只声明一个变量。
- `G.DCL.06`：使用字面量风格声明，例如 `{}` / `[]`。
- `G.TYP.01`：浮点数字面量不能写成 `.08` 或 `1.`。

处理建议：

- 用 Babel/parser 解析当前导出单里的 JS 文件，`sourceType` 优先设为 `unambiguous`，兼容脚本和模块。
- 只改当前导出单文件集合；改完后用 `git diff --name-only` 和导出单文件集合做精确对比。
- `var` 改 `const/let` 前先扫描后续赋值和自增自减；不确定时用 `let`。
- 多声明拆分时不要拆 `for` / `for-in` / `for-of` 初始化语句。
- `new Object()` / `new Array(a, b)` 可改 `{}` / `[a, b]`；`new Array(length)` 语义特殊，不要机械改成 `[length]`。
- 嵌套三元可以改成立即执行函数中的 `if/return`，这是表达形式整改；不要顺手重排周边控制流。
- `alert()` 不是纯文本替换。只有项目已有同语义通知入口且周边已有模式时，才替换为 `setStatus` / toast / message；否则归入风险桶。
- Babel generator 可能保留或重写 `NumericLiteral.extra.raw`，最终必须再扫源码文本或 AST raw，确认不存在 `.08` 这类省略 0 写法。

JS 批量改完至少验证：

- 所有目标文件 Babel parse 成功。
- 所有目标文件 `node --check` 成功。
- 专用规则反扫为 0：嵌套 `ConditionalExpression`、`var`、非 `for` 多声明、裸 `alert`、`new Object/Array`、省略 0 的浮点 raw。
- `git diff --check HEAD` 成功。
- 已修改的 tracked 文件集合等于当前导出单文件集合。

### 5. 有些规则看起来像风格问题，实际上是语义问题

尤其是：

- `protected-access`
- 返回值统一
- `try/except`
- `print -> logger`
- `SystemExit`
- `shell=True / shell=False`
- fallback / cache / history / runtime state

这些默认应该进“风险桶”，而不是“直接改”。

`G.FNM.07` 要特别保守。只有在“不改变调用次数、异常传播、返回契约”的前提下，才可以把裸调用改成显式接收并消费返回值，例如赋给 `_result` 后做无语义 no-op。不能为了规则改成新的分支、吞异常或改变失败路径。

### 6. 不要为了过规范去抽 helper 或整理结构

如果一个规则要靠下面这些动作才能通过：

- 抽局部 helper
- 拆大函数
- 重新组织控制流
- 整理返回结构

那它通常已经超出“代码表达形式整改”的边界。

默认做法应该是：

- 记录风险
- 停止自动推进
- 等用户确认

## 通用分桶法

### 低风险机械桶

- 文件头
- docstring
- 长行
- 未使用导入
- 注释废代码
- `staticmethod / classmethod`
- `G.CLS.06`
- JS 字面量、声明拆分、嵌套三元等纯表达规则
- 基础格式规则

### 高风险边界桶

- `G.CLS.11`
- `G.LOG.02`
- `G.ERR.07`
- `G.ERR.11`
- `alert` 在没有现成等价通知入口时
- fallback / cache bypass
- session / history / runtime scope
- 任何会改返回契约的规则
- 任何需要抽 helper、拆函数、整理结构的规则

## 推荐执行顺序

1. 解析导出单，统计规则和文件分布。
2. 先按风险分桶。
3. 只把低风险表达形式桶并行派发。
4. 主代理统一复核。
5. 第一轮结束后重新统计剩余问题。
6. 若后期主要剩 `G.CLS.06`，单开“纯方法重排轮”。
7. 若后期主要剩边界规则，停止自动推进，改成风险确认。

如果用户反馈“问题越来越多”，先不要急着继续乱改。优先检查：

- 当前新表是否只剩少数规则族；
- 新旧表是否都是历史快照；
- `G.CLS.06` 是否在不同轮次给出了相反顺序；
- 当前轮关系是否能形成无环图；
- 已处理项是否已经提交并推送到目标分支。

## 推荐验证闭环

每批次都固定做：

- `python -m py_compile <files>`
- `git diff --check -- <files>`
- 最小 import smoke
- 长行扫描
- 对当前导出单做规则专属复核，例如 `G.CLS.06` 关系扫描。
- 对机械重排做 AST hash / decorator 对比，证明没有改方法体。
- 对 JS 表达规则做 AST/文本反扫和 `node --check`。

不要因为“只是规范”就跳过验证。

常用本地验证组合：

- `python -m compileall -q <files>`
- `python -m pycodestyle --max-line-length=120 <files>`
- `python -m pylint --disable=all --enable=<本轮相关规则> <files>`
- `python -m mccabe --min 21 <files>`
- `node --check <files>`
- `git diff --check HEAD`

验证要以本轮涉及文件为主；如用户问“所有历史表”，再汇总历史表文件集合。历史表复核时要明确区分“当前最新表 0 失败”和“旧快照与新顺序冲突”。

## subagent 协作建议

### 研究型

只回答：

- 哪些规则是纯表达形式问题；
- 哪些规则已经超出表达形式边界；
- 哪些文件可以互不干扰地并行改。

### 实现型

只改指定白名单文件，只处理指定规则桶。

### 主代理

负责：

- 维护总清单；
- 复核 patch；
- 统一验证；
- 判断是否停手。

## 常见误区

### 误区 1

为了规范去顺手重构。

### 误区 2

看到 `R0914` 就直接拆大函数。

### 误区 3

旧 Excel 里有问题，就默认当前还存在。

### 误区 4

剩余问题少，就默认最后一轮更容易。

### 误区 5

把多轮 `G.CLS.06` 当成同一套稳定规则。

实际经验是：新一轮导出单可能以新的当前代码为基准生成新的局部关系，和旧表叠加会成环。正确做法是按当前轮精确关系收敛，并把旧表当背景。

### 误区 6

只靠 diff 肉眼判断“只改了顺序”。

大规模方法移动 diff 会很吵，必须用 AST hash / decorator 对比证明方法体没动。

### 误区 7

遇到不懂的华为规则就靠猜。

规则含义不确定时，先查华为 CodeArts Check 官方资料；官方资料不足以解释接入行为时，再看 Gitee、Jenkins 或其他样例，但要明确那只是样例做法，不是产品契约。

### 误区 8

JS 批量整改只看 AST，不看最终源码文本。

实际经验是：数字字面量等规则可能藏在 raw 文本里。即使 AST 上数值相同，也要检查最终源码里是否仍有 `.08` / `1.` 这类写法。

## 最后记住的 7 条

1. 当前代码永远优先于导出单。
2. 第一轮先打低风险表达形式规则。
3. 后期剩余问题更接近边界问题。
4. `G.CLS.06` 可以单独开轮处理。
5. 超出表达形式边界的规则默认只记风险，不自动整改。
6. JS 批量规则要用 AST 改、用 AST 加源码文本反扫验。
7. 这份技能的价值是给代理稳定流程，而不是堆产品背景介绍。

## 多轮 CodeCheck 实战闭环

一套稳定的闭环是：

1. 每来一张新表，先统计规则、文件、风险桶。
2. 若全是 `G.CLS.06`，解析精确关系，建 DAG，拓扑重排。
3. 不猜全局顺序，不按旧表回滚。
4. 重排前保存方法 AST hash 和装饰器。
5. 重排后验证当前表关系 0 失败。
6. 再验证方法 hash / 装饰器 0 变化。
7. 若是 JS 批量表，额外跑 Babel parse、`node --check`、规则反扫和文件集合对比。
8. 跑编译、格式、目标 pylint、复杂度和 `git diff --check`。
9. 提交前确认分支和远端一致，提交后再推送目标分支。
