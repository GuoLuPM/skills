# 子代理任务拆分

## 概览

这份技能不负责回答“要不要派子代理”。

它处理的是下一步：既然已经准备派发，就要把任务拆得更窄，让每个子代理只拿到一个明确问题、最小可用上下文，以及主代理真正能复核的输出。

## 什么时候用

- 你已经确定会起一个或多个子代理。
- 你需要把一个模糊任务拆成更小、可复核的子问题。
- 你准备写派发提示词，而不是把整仓上下文直接丢给子代理。
- 你想把只读研究任务并行拆开，但不想拆错边界。
- 你希望子代理返回证据，而不只是结论。

## 不适用

- 任务还停留在“去找问题”这种宽泛目标。
- 你还没有决定派发是否值得。
- 改动很小，主代理自己做更快。
- 主代理还没有能力独立复核结果。

## 核心方法

### 1. 按问题拆，不按目录拆

不要这样写：

- “看看这个模块。”
- “去这片区域找问题。”

要这样写：

- “只确认 `submitOrder()` 是否仍是唯一写入口。”
- “只确认局部刷新失败后是否会退回整页重载。”

每个子代理应该只回答一个问题，而不是巡逻一个目录。

### 2. 先给视角，再给文件

研究型任务最稳定的两种视角是：

- `业务逻辑`
  - 看主路径、唯一真值、主入口、下游消费链。
- `降级 / 兜底`
  - 看主路径、触发条件、降级路径、决策点、语义损失、可观测信号。

先说明这次从哪个视角看，再给文件。

### 3. 只给最小闭环上下文

优先给：

- 1 到 3 个核心文件
- 1 份规则或架构文档
- 1 份模块文档
- 1 个相关测试或入口

不要给：

- 整个模块目录
- 一堆“可能相关”的文件
- 没说明作用的文档清单

### 4. 明确写出不该扩展到哪里

子代理最常见的失控原因，不是看不懂目标，而是被默认允许“顺手再看一点”。

派发里一定要写清：

- 哪些相邻问题不在范围内
- 哪些文件或模块不能碰
- 这次是只读，还是只允许改哪些文件

### 5. 先要证据，再要判断

不要只要结论。

必须要求：

- 文件路径
- 函数、符号或入口定位
- 已确认事实
- `[假设]`

如果证据不足，子代理应该明确说“未确认”，而不是补猜。

### 6. 只并行独立的只读任务

适合并行：

- “主写入口在哪里？”
- “同一条链上有没有降级路径？”

不适合并行：

- 子任务 B 依赖子任务 A 的答案
- 两个实现型子任务会改同一批文件
- 两个子任务其实都在共同决定一个最终方案

### 7. 实现型任务必须给文件白名单

一旦进入实现阶段，就不要再复用研究型提示词。

实现型提示词必须明确：

- 具体目标
- 可写文件
- 禁止触碰的文件
- 一旦范围扩大就停止并返回的规则

### 8. 复核型任务只检查一个维度

复核型子代理不应该“顺手把所有东西再看一遍”。

要固定成一个维度：

- 是否符合计划
- 是否越过模块边界
- 是否覆盖验收标准
- 是否引入回归风险

## 提示词模板

### 研究型：业务逻辑

```md
任务类型：研究型
任务视角：业务逻辑
任务目标：
- 只确认 <单一主链问题>。

阅读范围：
- <规则或架构文档>
- <模块文档>
- <核心代码文件>

禁止扩展到：
- <相邻问题 / 不相关模块>

输出要求：
- 回答主路径、唯一真值、主收口点、消费链。
- 每个结论附文件路径与函数 / 符号证据。
- 不确定项标记为 [假设]。
```

### 研究型：降级 / 兜底

```md
任务类型：研究型
任务视角：降级 / 兜底
任务目标：
- 只确认 <单一降级问题>。

阅读范围：
- <降级规则文档>
- <模块文档>
- <核心代码文件>

禁止扩展到：
- <相邻问题 / 不相关模块>

输出要求：
- 回答主路径、触发条件、降级路径、决策点、语义损失。
- 说明是否存在可观测信号。
- 不确定项标记为 [假设]。
```

### 实现型

```md
任务类型：实现型
任务目标：
- 在既定方案下完成 <单一实现任务>。

只允许修改：
- <文件白名单>

禁止触碰：
- <文件黑名单>

验收标准：
- 满足 <功能 / 文档 / 测试> 验收项。
- 不顺手重构，不扩大范围。
- 如需扩大范围，先停止并返回主代理。
```

### 复核型

```md
任务类型：复核型
任务目标：
- 只复核 <计划符合性 / 模块边界 / 验收标准 / 回归风险>。

只允许读取：
- <文件 / 文档 / 测试>

输出要求：
- 明确通过项与未通过项。
- 每个问题附证据定位。
- 不新增需求，不直接补实现。
```

## 常见错误

- 开头先说“去看看”，而不是先给一个判断问题。
- 把业务逻辑、降级分析和实现方案混到同一个子代理里。
- 直接丢一个目录过去，而不是给最小上下文包。
- 没写不该扩展到哪里。
- 没有证据就接受结论。
- 让实现型子任务并行改同一片写入区域。

## 最后自检

派发前快速确认：

- 任务是不是一句单一问题。
- 视角是不是明确写出来了。
- 上下文是不是最小闭环。
- 不该扩展到哪里是不是写清楚了。
- 输出里是不是要求了事实、`[假设]` 和证据。
